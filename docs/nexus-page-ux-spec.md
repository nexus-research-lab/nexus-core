# Nexus 页面与交互规格

> 更新时间：2026-03-24
>
> 本文用于收敛当前产品页面的职责、内容、交互和视觉表达。
> 当前版本以“首页双态 + Contacts + Room”为准，不再继续强化独立的 `nexus` 编排页面。

## 1. 总体结构

当前产品只保留 3 类主页面与 1 类页面状态：

- `首页 Launcher`
- `首页 App 对话态`
- `Contacts`
- `Room`

其中：

- `App / Nexus` 是系统级 Agent
- 它不再作为独立的可视化编排页存在
- 它通过首页输入框触发，并在首页内进入持续对话态
- 全局只保留一条 `App conversation`

## 2. 产品原则

### 2.1 首页既是品牌入口，也是系统 Agent 的唤醒入口

首页的第一职责仍然是“开始协作”。

但首页不只是启动器。用户输入系统级组织意图后，首页会进入第二状态：`App 对话态`。这意味着系统级 Agent 并不是一个割裂页面，而是首页的自然延展。

### 2.2 App conversation 全局唯一

系统级 Agent 不需要像 room 一样管理多条 conversation。

当前原则：

- 只保留一条全局 `app conversation`
- 不显示 `App conversations` 列表
- 不做 `nexus` 最近会话切换
- 不做单独的 `nexus workspace`

### 2.3 Contacts 负责发现人，Room 负责协作

这两个页面必须长期稳定：

- `Contacts`：成员目录、能力识别、发起动作
- `Room`：实际协作流、上下文、推进状态

系统级 Agent 不能替代这两个页面的职责。

## 3. 页面一：首页 Launcher

### 3.1 页面目标

用户在数秒内完成以下任一动作：

- 找到一个成员开始 1v1
- 找到一个已有 room 继续协作
- 触发系统级 Agent，开始组织动作

### 3.2 首页要表达什么

首屏只需要表达：

- 品牌
- 强主标题
- 单个中心输入框
- 轻量的最近 room / 最近成员

不应该表达：

- 复杂系统状态
- 运行时统计
- 高密度卡片列表
- 编排后台感

### 3.3 首屏结构

首屏只保留一组单一构图：

- 品牌与标题
- 一个中心 launcher 输入框
- 输入框下方少量最近入口
- 周边漂浮成员与 room 提示

### 3.4 主交互

输入框不是聊天框，而是协作启动器。

它承载三类意图：

1. 找成员
2. 找 room
3. 输入系统级组织意图

但这三类意图不再通过显式模式 pill 切换。

正确交互是：

- 用户直接输入
- 系统识别意图
- 若识别为系统级意图，则进入首页 `App 对话态`

### 3.5 不该出现的元素

- `Agent / Room / Ask App` 模式切换
- 独立的 `Nexus` 管理面板入口
- 多条 App conversation 列表
- 复杂 filters / tabs / dashboard

### 3.6 视觉与动效

首页是品牌最强的页面。

应保持：

- 单一 hero 构图
- 全幅玻璃 blob 语言
- 强品牌感
- 高级但克制的动效

## 4. 页面二：首页 App 对话态

### 4.1 页面目标

当用户输入系统级组织意图时，首页进入系统 Agent 对话态，用同一张画布承载持续系统对话。

### 4.2 这个状态要表达什么

它要让用户感知到：

- 还是同一张首页
- 系统级 Agent 已被唤醒
- 现在正在和产品里的顶层管家对话

而不是：

- 打开了另一个后台
- 跳进了另一个产品子系统

### 4.3 页面结构

推荐布局：

- 左侧：收缩后的 Launcher
- 右侧：`App Conversation Panel`

左侧保留：

- 品牌
- 收缩后的输入入口
- 最近 room / 最近成员的轻量信息

右侧承载：

- `App Agent` 标题
- 唯一的全局对话流
- 系统级输入框

### 4.4 主交互

适合在这里完成的事情：

- 创建 room
- 建议应该邀请哪些成员
- 创建新成员
- 整理最近协作网络
- 将系统级组织结果导向一个具体 room

不适合：

- 在此长期承担业务协作
- 在此替代某个专业成员聊天
- 在此管理多条 conversation

### 4.5 动效要求

从首页启动态进入 `App 对话态` 时，必须有连续形变，而不是路由硬切。

建议至少包含：

1. hero 主构图向左收缩
2. 首页内容层级减弱
3. 右侧对话窗从右侧或玻璃雾层中展开

动效表达应是：

- 首页被唤醒
- 系统 Agent 被激活

而不是：

- 打开一个抽屉
- 跳转到另一个页面

### 4.6 工程状态建议

视觉上不独立成新页面，但工程上应可恢复状态。

建议形态：

- `/`：首页启动态
- `/?surface=app`：首页 App 对话态

这样既保留同页体验，也保留可恢复性。

## 5. 页面三：Contacts

### 5.1 页面目标

Contacts 是成员网络页，不是聊天页。

用户来这里是为了：

- 浏览成员
- 看成员擅长什么
- 发起 1v1
- 把成员带去协作

### 5.2 页面要表达什么

这里的核心不是“谁在线”，而是：

- 我现在有哪些数字成员
- 这些成员分别擅长什么
- 我下一步该跟谁协作

### 5.3 页面结构

建议两段式：

- 左侧：成员列表 / 搜索 / 轻筛选
- 右侧：成员 profile / 能力 / 主动作

### 5.4 主动作

每个成员至少有两个主动作：

- 发起 1v1：进入对应 room
- 带去系统组织：回到首页 `App 对话态` 或唤起 App Agent

### 5.5 不该出现的元素

- 长对话流
- room 级上下文
- 复杂运行面板

### 5.6 视觉气质

Contacts 应该更像成员目录与网络页：

- 更静
- 更清晰
- 更强调列表与 profile 的切换关系

## 6. 页面四：Room

### 6.1 页面目标

Room 是真正协作发生的地方。

### 6.2 Room 页面必须表达什么

用户必须一眼看懂：

- 自己在一个 room 里
- 这个 room 下有多个 conversations
- 当前正在推进哪条 conversation
- 谁在参与
- 协作推进到了哪里

### 6.3 固定信息结构

左侧第一层永远是：

- `Conversations`

而不是：

- `Rooms`

中间：

- 当前协作流
- room composer

右侧：

- 成员状态
- 推进状态
- 上下文边界

### 6.4 单成员 room

重点应放在：

- 当前对话
- 当前成员状态
- 当前上下文

右侧更像单个成员的协作状态区。

### 6.5 多成员 room

重点应放在：

- 成员分工
- 当前活跃成员
- room 级计划推进
- `@成员` 指派

右侧更像 room 级协作面板，而不是某一个成员面板。

### 6.6 输入区原则

room 输入区默认就是 room composer。

不应出现：

- `Agent`
- `Room`
- `Ask App`

这种模式切换。

用户既然已经进入 room，就不应再在输入区层面重新选择页面语义。

## 7. 页面间跳转关系

### 7.1 首页 -> App 对话态

当用户输入系统级组织意图时发生。

### 7.2 首页 -> Room

当用户直接选择某个成员、某个 room 或某条已有协作时发生。

### 7.3 Contacts -> Room

发起 1v1 协作。

### 7.4 Contacts -> 首页 App 对话态

当用户想把成员纳入一个更复杂的组织动作时发生。

### 7.5 App 对话态 -> Room

当系统级 Agent 完成组织动作，决定具体进入哪个协作空间时发生。

## 8. 当前实现上的落地顺序

建议按这个顺序推进：

1. 首页双态规格定稿
2. 首页 `App 对话态` 动效和布局实现
3. Contacts 继续做真实筛选与动作
4. Room 区分 1v1 和多人右侧结构

## 9. 验收标准

当以下条件成立时，当前页面层产品结构视为对齐：

- 不再强化独立的 `nexus` 编排页面
- 首页具备清晰的双态：`Launcher / App 对话态`
- 全局只有一条 `App conversation`
- Contacts 只负责成员网络与发起动作
- Room 只负责真实协作
- 输入区不再通过模式 pill 混合页面语义
