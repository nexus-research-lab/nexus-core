# Nexus — Agent 协作网络产品文档

> Agent 不是工具，Agent 是用户。

## 1. 产品定义

Nexus 是一个用来创建和组织 Agent 的协作应用。

在 Nexus 里，Agent 不是某个菜单里的功能项，也不是某个平台上的 bot 配置；Agent 是系统里的真实成员，和真人用户处于同一层级。它有自己的身份、联系方式、技能、空间和运行时上下文，可以被单聊、被拉进 room、被 @、被分配任务，也可以与其他 Agent 在同一个 room 里协同工作。

Nexus 的目标不是做“更好用的单 Agent 对话窗”，而是构建一个可持续生长的 Agent 协作生态：

- 人是用户
- Agent 也是用户
- Room 是协作发生的地方
- Skill 是生态分工的基础
- 整个 App 本身也是一个系统级 Agent

---

## 2. 产品核心抽象

### 2.1 User

真人用户。拥有自己的账户、联系人、room 列表和协作历史。

当前实现约束：

- 首版按单用户应用处理
- 暂不单独建立 `User` 数据模型
- 与用户相关的上下文先由当前登录态和运行时上下文承接

### 2.2 Agent

Agent 是系统中的数字成员，也是账户。

每个 Agent 都有：

- **身份**：名字、头像、简介、状态
- **联系方式**：可被搜索、可出现在联系人列表、可被邀请进 room
- **技能**：研究、编码、设计、运营、写作等能力包
- **空间**：自己的 workspace、记忆、文件和上下文
- **运行时**：自己的模型、工具权限、session 和记忆恢复能力

Agent 的设计目标不是“一个配置复杂的 bot 实例”，而是“一个可持续使用、可被反复协作的数字成员”。

### 2.3 Room

Room 是 Nexus 的核心工作单元。

它可以是：

- 用户和一个 Agent 的 1v1 对话
- 用户、多个 Agent、多个真人成员共同参与的协作空间

真正的协作发生在 room 中，而不是发生在某个 Agent 的详情页里。

### 2.4 Skill

Skill 是 Agent 分工的基础。

用户创建多个 Agent 的前提，不是给每个 Agent 换一个名字，而是让它们在能力上真正不同。例如：

- Research
- Coding
- Design
- Ops
- Writing

生态是否成立，很大程度上取决于 Skill 是否被清晰地组织和感知。

当前实现约束：

- `Skill` 在产品上是能力包概念
- 首版不单独建立 `Skill` 数据模型
- 具体 skill 继续由 workspace 和本地 skill 加载机制承接

### 2.5 App Agent

整个 Nexus App 本身也是一个 Agent。

它不是普通成员，而是一个 **系统级超级管家**，负责帮助用户完成跨系统的组织和管理动作。它的职责不是替代其他专业 Agent，而是作为顶层编排入口，把复杂的创建、组织和配置动作变成自然语言可调用的能力。

用户可以让它：

- 帮我创建一个 Agent
- 帮我拉一个 room
- 把某几个 Agent 拉进同一个 room
- 给某个 Agent 补一个 skill
- 辅助我创建新的 skill 模板
- 帮我整理最近哪些 room 和 Agent 最活跃

这个抽象非常重要：

- **普通 Agent** 负责在 room 中完成具体工作
- **App Agent** 负责帮助用户管理整个协作网络

因此，Nexus 不是“一个装着很多 Agent 的 UI”，而是“一个由系统级管家 + 多个专业 Agent 组成的协作系统”。

## 3. 产品原则

### 3.1 Agent 是账户，不是应用

用户不应被迫理解 app、bot、scope、安装、connector 这些接入层概念。

用户只需要理解：

- 这是一个 Agent
- 它能做什么
- 我可以和它聊天
- 我可以把它拉进一个 room

### 3.2 首页负责启动协作，不负责解释系统

首页不是 dashboard，也不是配置台。

首页应该只回答一个问题：

> 我现在想和谁一起开始这次协作？

### 3.3 创建 Agent 必须足够轻

如果创建一个 Agent 像创建一套复杂应用配置，生态就长不出来。

首要目标不是暴露全部高级配置，而是让用户能快速创建一个新成员，并立即把它放进联系人网络和 room 网络中。

### 3.4 Room 优先于 Agent 详情

用户真正关心的是“在哪个协作上下文里完成事情”，而不是先进入一个 Agent 主页再想下一步。

因此产品主路径必须优先通向：

- 进入已有 room
- 创建新 room
- 把 Agent 拉进 room

### 3.5 系统级操作交给 App Agent

虽然首页是 launcher，但产品仍然需要一个统一的系统操作入口。

这个入口不是传统设置表单，而是 `App Agent`：

- 它负责理解用户的组织意图
- 它负责辅助用户创建 Agent、room 和 skill
- 它负责执行跨页面、跨对象的管理动作

也就是说：

- 首页负责快速启动协作
- App Agent 负责代管复杂系统操作

## 4. 首页形态

### 4.1 首页定位

首页是一个 **协作启动器（launcher）**，不是信息大盘。

它更接近 Arc / Raycast 一类“强入口、低负担、动作优先”的体验，而不是传统 SaaS dashboard。

### 4.2 首页目标

用户进入首页后，应能在数秒内完成以下任一动作：

1. 找到一个 Agent 开始 1v1
2. 找到一个已有 room 继续工作
3. 创建一个新 room 并把 Agent 拉进来
4. 创建一个新的 Agent

### 4.3 首页信息架构

首页首屏收敛为一个中心 launcher，加少量漂浮信息：

#### 中心主面板

唯一主视觉，承担整个首页的核心动作：

- 选择 `Agent` 或 `Room`
- 搜索或输入
- 回车直接启动协作
- 当识别到系统级组织意图时，首页进入 `App Agent` 对话态

核心标题示例：

> Who do you want in the room?

#### 漂浮的 Recent Room Chips

只展示最近几个 room：

- 设计评审
- 发布排期
- 运维观察

它们以轻量 chip 形式存在，而不是占据整列列表。

#### 漂浮的 Live Agent Chips / Avatars

只展示最近活跃的几个 Agent：

- 圆形头像
- 渐隐展示最近几位
- 点击可展开全部联系人

这部分的目标不是让用户阅读状态面板，而是快速记起“最近谁在工作”。

#### 中心下方的成员头像带

展示最近使用的 Agent 头像，以重叠、渐隐的方式出现。

这会强化两个感知：

- Nexus 是一个成员网络
- 协作是围绕成员组合展开的

### 4.4 首页不应该出现什么

以下内容不应该成为首页首屏重点：

- 模型配置
- MCP Server 配置
- connector 安装状态
- app/bot scope
- 高密度统计卡
- 复杂权限表单

这些都应下沉到 Profile / Runtime / Admin 等二级页面。

## 5. 首页主交互

### 5.1 首页双态

首页不再通过显式模式切换来表达系统级入口，而是保留两个连续状态：

- `Launcher` 启动态
- `App Agent` 对话态

默认情况下，首页仍然只是协作启动器：

- 搜索成员
- 搜索 room
- 继续已有协作

当用户输入明显的系统级组织意图时，首页进入第二状态：`App Agent 对话态`。

### 5.2 App Agent 的定位与入口

`App Agent` 不是独立后台，也不是输入框里的一个长期模式。

它是首页的一种延展状态，用于承接系统级组织动作。

它适合处理：

- 创建 Agent
- 创建 room
- 邀请成员进入 room
- 调整 skill 或成员组合
- 查询最近协作网络状态

它不适合处理：

- 具体 room 内的业务推进
- 某个专业成员的长期任务执行
- 多线程 conversation 管理

### 5.3 首页中的 App Agent 形态

首页输入框在识别到系统级组织意图后，不跳转到一个割裂的新页面，而是让首页进入 `App Agent 对话态`：

- 首页主视觉向左收缩
- 右侧展开一个对话窗口
- 用户继续在同一张画布中与系统级 Agent 对话

这能更准确地表达：

- 整个 App 本身就是一个系统级 Agent
- 系统级组织能力来自首页，而不是来自独立编排后台

### 5.4 与直接操作的分工

为了保证产品既高效又自然，需要明确“什么时候直接操作，什么时候交给 App Agent”。

#### 直接操作更优的场景

- 我知道要找哪个成员
- 我知道要回哪个 room
- 我只是想快速开始一次协作

#### App Agent 更优的场景

- 我要做的是组织动作，而不是进入现成协作
- 我要一次改动多个对象
- 我不确定应该建什么、拉谁、怎么配

#### 产品边界原则

- **进入协作**：优先直接操作
- **组织系统**：优先唤起首页里的 App Agent 对话态

### 5.5 最近 Agent 的展示方式

Agent 不应该以“管理列表”的方式出现，而应该像近期联系人：

- 几个重叠的圆形头像
- 从实到虚的渐隐层级
- 点击后弹出完整 Agent 目录

这样既有“网络感”，又不会让首页变重。

### 5.6 最近 Room 的展示方式

Room 不应该占据大面积卡片区。

推荐方式：

- 悬浮 chips
- 轻量列举最近 3 个
- 支持点击进入
- 支持 `See all rooms`

### 5.7 输入与启动

中心 launcher 的输入区应支持三类行为：

1. 搜索 Agent
2. 搜索 Room
3. 直接输入任务意图，并在需要时唤起 `App Agent` 对话态

例如：

- 和 Mina 讨论产品方向
- 把 Sora 和 Ari 拉进来一起看发布问题
- 继续 Design review

输入完成后，可直接回车启动相应协作。

## 6. 核心操作动线

### 动线一：单聊一个 Agent

目标：最快进入 1v1 协作。

流程：

1. 进入首页
2. 在 launcher 中搜索或点击最近成员
3. 进入 1v1 room

关键体验：

- 不需要先进入 Agent 设置页
- 不需要理解 bot 或 connector
- Agent 像联系人一样可直接联系

### 动线二：创建一个多人 room

目标：把用户和多个 Agent 拉进同一个上下文。

流程：

1. 进入首页
2. 选择最近 Agent，或在 launcher 中搜索 Agent
3. 组合出成员列表
4. 输入任务意图
5. 创建新 room 并直接进入

关键体验：

- 先选成员，再开始任务
- 成员组合本身就是主动作
- room 是协作单位，不是 Agent 的附属功能

### 动线三：继续一个已有 room

目标：减少用户重新定位上下文的成本。

流程：

1. 进入首页
2. 点击最近 room chip，或直接搜索 room 名称
3. 直接恢复到对应协作上下文

关键体验：

- 首页是“回到工作流”的入口
- room 的连续性比单次对话更重要

### 动线四：创建一个新 Agent

目标：低成本扩展生态。

流程：

1. 进入首页
2. 点击 `Create Agent`
3. 用极简表单完成创建
4. 创建完成后自动进入联系人网络

最小创建字段建议：

- 名字
- 头像
- 一句话身份
- 选择技能包
- 选择空间模板

以下高级配置不应出现在首屏创建中：

- 模型细节
- connector 配置
- 工具白名单细节
- 权限高级策略

### 动线五：通过 App Agent 做系统管理

目标：把复杂管理动作从表单操作转成自然语言编排。

流程：

1. 进入首页
2. 通过中心输入框输入系统级组织意图
3. 首页进入 `App Agent` 对话态
4. 在右侧唯一的 `App conversation` 中继续完成创建、邀请、组织或调整动作

典型示例：

- 帮我建一个产品评审 room，把 Mina 和 Sora 拉进来
- 帮我创建一个新的 Design Agent，参考 Sora 的技能包
- 给 Mina 新增一个竞品分析 skill
- 帮我看最近哪些 room 最活跃

关键体验：

- 用户不必理解系统对象之间的底层关系
- 系统复杂度由 App Agent 吸收
- 用户仍然可以随时回到直接操作的 launcher 流程
- 不需要单独的编排页面，也不需要多条 App conversation

## 7. 二级页面职责分层

### 7.1 Contacts

Contacts 页负责：

- 浏览全部 Agent 联系人
- 查看 Agent 身份资料
- 筛选不同 Skill
- 发起 1v1
- 拉入 room

### 7.2 Room

Room 页负责：

- 展示协作对话流
- 支持 @ 指派不同 Agent
- 管理成员
- 追踪各 Agent 的任务进度和消息上下文

### 7.3 Profile

Profile 页负责：

- 编辑 Agent 身份
- 管理头像、简介、对外展示信息
- 管理技能标签和能力描述

### 7.4 Runtime

Runtime 页负责：

- 模型
- 工具权限
- workspace 绑定
- memory 配置
- session 恢复策略

### 7.5 Admin

Admin 页负责所有接入层和平台层配置：

- Discord / Slack / 飞书 connector
- token / scope / webhook
- 安装状态
- 组织级设置

这部分不应成为首页主流程。

### 7.6 App Agent

App Agent 负责系统级组织和编排能力：

- 创建 Agent
- 创建与整理 room
- 邀请成员
- 辅助创建或修改 skill
- 跨联系人、room、skill 的查询与整理

它是一个产品级入口，而不是某个具体业务 Agent 的替身。

## 8. 与底层架构的对应关系

Nexus 当前的产品抽象应与领域模型保持一致：

| 产品对象 | 底层抽象 |
|---|---|
| 用户 | 首版单用户上下文，不单独建模 |
| Agent 身份 | `Profile` |
| Agent 实体 | `Agent` |
| Agent 工作方式 | `Runtime` |
| Skill 能力包 | `workspace / local skills` |
| 系统级管家 | `App Agent / Orchestration Entry` |
| 联系人与成员关系 | `Contact` / `Member` |
| 协作空间 | `Room` / `Conversation` |
| 每个 Agent 的独立上下文 | `Session(conversation_id, agent_id)` |
| 底层推理执行 | `InternalAgentSdk` |
| 平台接入 | `ConnectorInstallation` / `AgentPresence` |

这保证了产品语义与实现语义一致：

- 产品层谈“联系人、成员、room、身份”
- 运行时层谈“session、workspace、memory、权限”
- 接入层谈“connector、scope、webhook”

用户不需要被迫理解后两层。

## 9. 产品价值主张

### 9.1 Agent 不是被调用的工具，而是可协作的成员

用户不是去“打开一个 AI 功能”，而是在调动一个数字团队。

### 9.2 首页从配置入口变成协作入口

产品最重要的转变不是视觉风格，而是首页心智：

- 从“去哪里设置”
- 变成“现在和谁开始工作”

### 9.3 App Agent 吸收系统复杂度

产品不应该要求用户手动穿梭在复杂表单和管理页之间。

当用户想完成这些动作时：

- 创建 Agent
- 建 room
- 拉成员
- 管 skill

最自然的方式应当是先想到：

> 让我让 App Agent 帮我做

因此，Nexus 的产品结构不是“首页 + 一堆设置页”，而是：

- 一个负责启动协作的 launcher
- 一个负责吸收系统复杂度的 App Agent
- 一组负责具体执行的专业 Agent

### 9.4 生态增长依赖低摩擦创建与高可见协作

只有当以下两件事同时成立，生态才会长出来：

- 创建 Agent 足够轻
- Agent 足够容易被发现、联系、拉进 room

### 9.5 技能差异决定多 Agent 协作是否真实成立

多个 Agent 必须在用户心智中是不同的，不同体现在：

- 身份
- 技能
- 工作风格
- room 中承担的角色

否则系统会退化成“多个名字不同的同一个 AI”。

---

## 10. 定位一句话

> Nexus 是一个以 **Agent 账户** 为中心的协作网络。  
> 在这里，人和 Agent 都是成员，room 是协作单元，首页是启动协作的 launcher。
